## 从 demo 到生产

过去两年我参与了几个 LLM 应用项目，从最初简单的"调 API 做个聊天机器人"，到后来涉及 RAG、多 Agent 协作的复杂系统。技术选型是贯穿始终的难题——AI 领域变化太快，今天的最佳实践可能下个月就过时了。

这篇文章整理一下我在技术选型上的思考框架。

### 模型选型：能力 vs 成本

模型选择是 LLM 应用最核心的决策。我通常从三个维度评估：

**1. 能力门槛**

你的场景需要多强的模型？举个例子：

- **客服/FAQ 问答**：GPT-4o-mini 或 Claude Haiku 就足够，成本低、速度快
- **代码生成/复杂推理**：需要 Claude Opus 或 GPT-4 级别的模型
- **结构化数据提取**：中等模型 + 好的 prompt 通常可行

**2. 延迟要求**

- 实时聊天：首 token 延迟必须在 500ms 以内
- 离线批处理：延迟不重要，成本更重要
- Streaming 是必须的——几乎所有生产场景都需要

**3. 成本预算**

一个简单的估算公式：

```
每日成本 = 请求数 × (输入 token + 输出 token) × 每 token 单价
```

对于中等规模应用（每日 10 万次请求），API 调用成本可能从几百到几千美元不等。这时候自部署开源模型（如 Llama、Qwen）就成为一个选项。

### RAG 架构的选择

RAG（检索增强生成）是当前最实用的 LLM 应用模式。选型时我关注这几个组件：

#### 向量数据库

| 方案 | 适用场景 | 部署方式 |
|------|----------|----------|
| Milvus/Zilliz | 大规模（百万级向量）、高并发 | 自部署/云 |
| Weaviate | 中等规模，需要多模态 | 自部署/云 |
| pgvector | 小规模，已有 PostgreSQL | 嵌入现有数据库 |
| Chroma | 原型开发 | 本地/嵌入式 |

对于大多数项目，我建议从 **pgvector** 开始——如果你已经在用 PostgreSQL，零额外运维成本。当向量规模超过百万级，再迁移到专门的向量数据库。

#### Embedding 模型

- 中文场景优先考虑 **BGE** 或 **Qwen** 系列
- 多语言场景用 **text-embedding-3** 系列
- 考虑使用 embedding 缓存来降低 API 成本

### Agent 框架的选择

市面上 Agent 框架很多，我的建议是：

- **LangChain**：生态最丰富，但抽象层太多，调试困难。适合快速原型，生产环境需要谨慎
- **LlamaIndex**：数据索引和处理更强，适合 RAG 密集型场景
- **Vercel AI SDK** 或 **DIY**：对于简单的 Agent 逻辑，自己写反而更可控

实际项目中，我倾向于"**组件化引用**"的态度——用框架的某个模块（比如 LangChain 的 document loader），但核心的 Agent 逻辑自己实现。

### 前端技术栈

LLM 应用的前端有一些特殊需求：

- **Streaming 响应**：使用 Server-Sent Events（SSE）或 WebSocket。Next.js 的 Streaming SSR 和 Server Actions 很适合这个场景
- **Markdown/富文本渲染**：LLM 输出通常是 Markdown，用 `react-markdown` 或 `next-mdx-remote` 渲染
- **状态管理**：聊天历史用 React Context + localStorage 就够，不需要复杂的状态管理库

工具链推荐：

```
前端框架： Next.js / Nuxt.js
UI 组件： shadcn/ui / Tailwind CSS
流式渲染： Vercel AI SDK / 自建 SSE
数据持久化： PostgreSQL + Drizzle ORM
```

### 总结

技术选型没有银弹。我的原则是：

1. **能不引入新组件就不引入**——每个依赖都是未来的技术债务
2. **从简单开始**——先用最直接的方式验证方案可行性
3. **关注解耦**——LLM 相关的逻辑和业务逻辑要分开，方便后续替换模型

最终，技术选型是一个持续迭代的过程，不是一次性的决策。
